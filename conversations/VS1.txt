Feasibility and recommendations (short, actionable)

- Is Proton Drive-only possible?
  - Likely yes, but depends on Proton Drive API capabilities and auth model. Two important constraints:
    - Server-side unattended sync requires Long-lived server credentials (API token or service account) or an OAuth2 refresh flow you can run on the server. If Proton only exposes short-lived user OAuth tokens without refresh capability or no programmatic file API, implementing unattended server sync becomes hard.
    - Proton emphasizes privacy and end-to-end encryption. If Proton Drive encrypts client-side and does not provide a normal REST file API for third-party servers, integration will be more work or impossible without using Proton's own client/CLI or reverse-engineered APIs.
  - Practical suggestion: prefer Proton for privacy if the Proton Drive API supports server tokens or a CLI that can be scripted; otherwise use Google Drive (service accounts) for robust, server-side, unattended backups.

- Google Drive alternative
  - Google Drive has mature APIs and supports server-side service accounts and granular access control ‚Äî easiest for unattended programmatic uploads and listing.
  - It‚Äôs less privacy-by-default compared with Proton, but you can encrypt content before upload (zero-knowledge) if you want privacy and still use Drive‚Äôs robust API.

Trade-offs (short)
- Proton
  - Pros: Privacy-focused, good for a project aiming to be independent of big clouds.
  - Cons: API maturity and unattended-server support may be limited; may require manual token setup or CLI use; could need more engineering to automate.
- Google Drive
  - Pros: Stable API, service accounts, better tooling for server-side automation.
  - Cons: Less privacy by default ‚Äî mitigate by encrypting content prior to upload.
